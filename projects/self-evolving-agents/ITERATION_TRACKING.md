
# 迭代追踪机制 - 连续反思链

## 目标
建立完整的迭代追踪体系，确保每次迭代都基于前一天的反思，形成连续的改进链。

## 迭代连续性原则

### 1. 每个迭代都有"根基"
- 每个迭代必须明确标注"基于哪几天的反思"
- 每个迭代必须记录"解决了之前识别的哪个问题"
- 每个迭代必须与之前的迭代形成逻辑链条

### 2. 改进点状态追踪
- 每个识别的改进点都有状态：未开始 → 进行中 → 已完成 → 已验证
- 改进点不能"消失"，必须有明确的结案
- 长期追踪改进点的实际效果

### 3. 行动项完成度追踪
- 每个行动项都有完成度：0% → 50% → 100%
- 未完成的行动项必须说明原因
- 未完成的行动项必须滚动到次日计划

## 迭代追踪文档结构

### 迭代元数据
```markdown
## 迭代基础信息
- **迭代编号**: Iteration XXX
- **创建日期**: YYYY-MM-DD
- **基于的反思**: [日期1]、[日期2]
- **解决的问题**: [描述之前识别的具体问题]
- **前序迭代**: Iteration XXX → Iteration XXX → 当前迭代
```

### 改进点状态表
| 改进点 | 提出日期 | 当前状态 | 预计完成 | 备注 |
|--------|----------|----------|----------|------|
| [改进点1] | [日期] | [状态] | [日期] | [备注] |
| [改进点2] | [日期] | [状态] | [日期] | [备注] |

### 迭代效果评估
```markdown
## 迭代效果评估
- **预期效果**: [描述预期达到的效果]
- **实际效果**: [描述实际达到的效果]
- **效果差异**: [分析差异原因]
- **后续调整**: [描述基于效果的后续调整]
```

## 反思连续性检查清单

每次反思前，必须检查：

- [ ] 是否回顾了前一日的反思？
- [ ] 是否评估了前一日改进点的状态？
- [ ] 是否检查了前一日行动项的完成情况？
- [ ] 是否从前一日反思中获得了持续洞察？
- [ ] 本次反思是否与前一日形成了逻辑链条？

每次迭代前，必须检查：

- [ ] 本次迭代是否明确基于之前的反思？
- [ ] 本次迭代是否解决了之前识别的具体问题？
- [ ] 本次迭代是否与之前的迭代形成了连续链？
- [ ] 是否记录了迭代的预期效果？
- [ ] 是否建立了迭代效果的评估方法？

## 迭代链可视化

### 示例迭代链
```
Day 1 反思 → 识别问题A、B、C
    ↓
Day 2 反思 → 回顾Day 1 → 验证问题A → 迭代解决问题B
    ↓
Day 3 反思 → 回顾Day 2 → 验证问题B解决 → 迭代解决问题C
    ↓
Day 4 反思 → 回顾Day 3 → 验证问题C解决 → 识别新问题D、E
    ↓
...持续进化...
```

## 使用流程

### 每日反思流程
1. 打开前一日的反思文件
2. 回顾前一日识别的改进点，更新状态
3. 检查前一日行动项的完成情况
4. 从前一日反思中提取持续洞察
5. 进行当日的正常反思
6. 记录本次反思基于的历史反思
7. 更新改进点状态追踪表

### 迭代实施流程
1. 从连续反思中识别真正需要迭代的问题
2. 明确本次迭代基于哪几天的反思
3. 记录本次迭代解决的具体问题
4. 建立与前序迭代的逻辑链条
5. 设定迭代的预期效果
6. 建立迭代效果的评估方法
7. 实施迭代
8. 评估迭代效果
9. 更新迭代追踪文档
