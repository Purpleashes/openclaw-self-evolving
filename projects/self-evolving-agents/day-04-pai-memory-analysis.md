## 1. 项目概述

### PAI 是什么？
- **Personal AI Infrastructure** - 个人人工智能基础设施
- **核心理念**: AI应该放大每个人的能力，而不仅仅是少数人
- **目标**: 激活人们的创造力，帮助人们识别、表达和追求自己的人生目标

### 三个层次的AI系统
PAI将AI系统分为三个层次：

1. **聊天机器人** (Chatbots)
   - 模式：提问 → 回答 → 忘记
   - 例子：ChatGPT, Claude, Gemini
   - 问题：没有记忆，每次对话都从头开始

2. **代理平台** (Agentic Platforms)
   - 模式：提问 → 使用工具 → 获取结果
   - 例子：Claude Code
   - 问题：更有能力，但仍然不了解你

3. **PAI** (Personal AI Infrastructure)
   - 模式：观察 → 思考 → 计划 → 执行 → 验证 → **学习** → 改进
   - 特点：
     - 捕获每个信号（评分、情感、验证结果）
     - 从错误中学习
     - 随着时间推移变得更好
     - 自我升级（技能、工作流、甚至核心行为）
     - 了解你的目标、偏好、历史

## 2. PAI 核心原则（16条）

| # | 原则 | 摘要 |
|---|------|------|
| 1 | **用户中心** | PAI围绕你构建，而不是工具。你的目标、偏好和上下文优先。 |
| 2 | **基础算法** | 科学方法作为通用问题解决循环：观察→思考→计划→构建→执行→验证→学习。 |
| 3 | **先清晰思考** | 好的提示来自清晰的思考。在写提示之前先澄清问题。 |
| 4 | **脚手架 > 模型** | 系统架构比使用哪个模型更重要。 |
| 5 | **确定性基础设施** | AI是概率性的；你的基础设施不应该是。使用模板和模式。 |
| 6 | **代码优先于提示** | 如果你能用bash脚本解决，就不要用AI。 |
| 7 | **规范/测试/评估优先** | 在构建之前编写规范和测试。衡量系统是否工作。 |
| 8 | **UNIX哲学** | 做好一件事。使工具可组合。使用文本接口。 |
| 9 | **ENG/SRE原则** | 像对待生产软件一样对待AI基础设施：版本控制、自动化、监控。 |
| 10 | **CLI作为接口** | 命令行界面比GUI更快、更可脚本化、更可靠。 |
| 11 | **目标→代码→CLI→提示→代理** | 决策层次：澄清目标，然后是代码，然后是CLI，然后是提示，然后是代理。 |
| 12 | **技能管理** | 模块化能力，根据上下文智能路由。 |
| 13 | **记忆系统** | 一切值得知道的都被捕获。历史为未来提供上下文。 |
| 14 | **代理个性** | 不同的工作需要不同的方法。具有独特声音的专业代理。 |
| 15 | **科学作为元循环** | 假设→实验→测量→迭代。 |
| 16 | **允许失败** | 明确允许说"我不知道"可以防止幻觉。 |

## 3. PAI 核心原语（Primitives）

PAI的原语是使其工作的核心系统架构。

### 3.1 TELOS（深度目标理解）

**10个文件捕获你是谁：**
- `MISSION.md` - 使命
- `GOALS.md` - 目标
- `PROJECTS.md` - 项目
- `BELIEFS.md` - 信念
- `MODELS.md` - 模型
- `STRATEGIES.md` - 策略
- `NARRATIVES.md` - 叙事
- `LEARNED.md` - 学到的东西
- `CHALLENGES.md` - 挑战
- `IDEAS.md` - 想法

**借鉴意义**：
- 我们已有类似的文件：`IDENTITY.md`, `SOUL.md`, `USER.md`
- 可以考虑扩展到更结构化的目标、项目、信念等文档
- 这种结构化的自我认知文档让AI真正"了解"用户

### 3.2 用户/系统分离

- **用户自定义内容** 放在 `USER/`
- **PAI基础设施** 放在 `SYSTEM/`
- 升级时，用户文件不受影响
- 可移植的身份，升级安全

**借鉴意义**：
- 我们当前的结构是混合的
- 可以考虑将用户配置和系统文件明确分离
- 便于备份、迁移和升级

### 3.3 细粒度自定义

**六个层次的自定义：**
1. **身份** - 姓名、声音、个性
2. **偏好** - 技术栈、工具
3. **工作流** - 技能如何执行
4. **技能** - 存在哪些能力
5. **钩子** - 事件如何处理
6. **记忆** - 捕获什么

**借鉴意义**：
- 我们当前只有部分层次（身份、部分偏好）
- 可以考虑建立更完整的自定义层次体系

### 3.4 技能系统

- 高度关注一致结果
- 结构：`CODE → CLI-BASED-TOOL → PROMPT → SKILL`
- 确定性结果优先

**借鉴意义**：
- 我们的技能系统相对简单
- 可以考虑更结构化的技能架构
- 强调代码优先，提示作为最后手段

### 3.5 记忆系统（核心重点）

**核心特点：**
- 专注于持续学习
- 每次交互产生信号：评分、情感、成功、失败
- 这些信号反馈到系统中改进系统
- **三层架构**（hot/warm/cold）
- 基于阶段的学习目录

**这是我们最需要借鉴的部分！**

### 3.6 钩子系统

**响应生命周期事件：**
- 会话开始
- 工具使用
- 任务完成
- 等等...

**8种事件类型支持：**
- 语音通知
- 自动上下文加载
- 会话捕获
- 安全验证
- 可观察性

**借鉴意义**：
- 我们可以考虑在关键节点添加钩子机制
- 例如：会话开始时自动加载相关记忆，任务完成时自动记录反思

### 3.7 安全系统

- 默认定义系统和用户级安全策略
- 安全钩子在执行前验证命令
- 阻止危险操作，同时允许正常工作流顺利进行

### 3.8 通知系统

- 通过ntfy推送通知进行移动提醒
- Discord集成用于团队更新
- 持续时间感知路由，对长时间运行的任务进行升级
- 即发即弃设计，通知永远不会阻塞工作流

### 3.9 语音系统

- 由ElevenLabs TTS提供动力
- 听到任务完成、会话摘要和重要更新的语音
- 韵律增强使语音听起来自然

### 3.10 基于终端的UI

- 丰富的选项卡标题和窗格管理
- 动态状态行显示学习信号、上下文使用情况和当前任务状态

## 4. 记忆系统深度分析

### 4.1 记忆系统的定位

PAI的记忆系统是其16条核心原则之一（原则13），也是其核心原语之一。

> **原则13：记忆系统** - 一切值得知道的都被捕获。历史为未来提供上下文。

### 4.2 记忆系统的核心功能

从README中提取的关键信息：

1. **持续学习**
   - 每次交互都会产生信号
   - 信号类型：评分、情感、成功、失败
   - 这些信号反馈到系统中，用于改进系统

2. **三层架构**
   - **Hot Memory**（热记忆）- 即时访问，当前会话
   - **Warm Memory**（温记忆）- 近期访问，最近几天/周
   - **Cold Memory**（冷记忆）- 长期存储，归档

3. **基于阶段的学习目录**
   - 记忆按照学习阶段组织
   - 便于回顾和迭代

### 4.3 与我们当前系统的对比

| 特性 | 我们的系统 | PAI系统 | 差距 |
|------|-----------|---------|------|
| 记忆分层 | P0/P1/P2（基于时间） | Hot/Warm/Cold（基于访问频率） | 不同的分层策略，可以互补 |
| 持续学习信号 | 手动记录反思 | 自动捕获评分、情感、成功、失败 | 我们缺少自动信号捕获 |
| 记忆检索 | memory_search（语义搜索） | 未明确说明，但可能有类似机制 | 我们的语义搜索是优势 |
| 用户认知文档 | IDENTITY.md, SOUL.md, USER.md | TELOS（10个文件） | PAI更全面 |
| 钩子机制 | 无 | 有（8种事件类型） | 我们缺少事件驱动的钩子 |
| 反馈循环 | 手动反思 | 自动反馈改进系统 | PAI的自动化更强 |

## 5. 我们可以借鉴的技术

### 5.1 信号捕获机制

**PAI的做法：**
- 每次交互后捕获信号
- 信号类型：
  - 评分（用户满意度）
  - 情感（正面/负面）
  - 成功/失败（任务是否完成）
  - 验证结果（输出是否正确）

**我们可以借鉴：**
1. **在任务完成后添加简单的反馈收集**
   - 例如："这个结果满意吗？👍/👎"
   - 自动记录到记忆中

2. **自动捕获成功/失败信号**
   - 工具执行成功/失败
   - 任务完成状态
   - 自动记录

3. **情感分析（可选）**
   - 对用户消息进行简单的情感分析
   - 记录情感趋势

### 5.2 三层记忆架构的补充

**PAI的三层：** Hot/Warm/Cold（基于访问频率）

**我们的三层：** P0/P1/P2（基于时间重要性）

**可以结合：**
- 保持我们的P0/P1/P2时间分层
- 添加访问频率追踪
- 高频访问的P1/P2可以"晋升"到更热的状态
- 低频访问的可以"降级"

### 5.3 TELOS用户认知文档

**PAI有10个文件：**
MISSION.md, GOALS.md, PROJECTS.md, BELIEFS.md, MODELS.md, STRATEGIES.md, NARRATIVES.md, LEARNED.md, CHALLENGES.md, IDEAS.md

**我们当前有：**
IDENTITY.md, SOUL.md, USER.md

**可以扩展：**
1. **GOALS.md** - 明确的短期和长期目标
2. **PROJECTS.md** - 当前和计划中的项目
3. **LEARNED.md** - 结构化的经验教训（类似我们的lessons，但更全面）
4. **BELIEFS.md** - 用户的核心信念和价值观

### 5.4 钩子系统

**PAI的8种事件类型：**
- 会话开始
- 工具使用
- 任务完成
- 等等...

**我们可以添加：**
1. **会话开始钩子**
   - 自动加载相关记忆
   - 回顾最近的任务
   - 问候用户

2. **任务完成钩子**
   - 自动记录反思
   - 询问用户反馈
   - 更新相关记忆

3. **工具使用钩子**
   - 记录工具使用情况
   - 分析工具使用模式
   - 优化工具推荐

### 5.5 用户/系统分离

**PAI的做法：**
- USER/ - 用户自定义内容
- SYSTEM/ - PAI基础设施
- 升级时用户文件不受影响

**我们可以借鉴：**
- 重新组织目录结构
- 将用户配置和系统文件明确分离
- 便于备份、迁移和升级

## 7. 总结

### PAI记忆系统的核心优势

1. **持续学习闭环** - 每次交互都产生信号，反馈改进系统
2. **结构化的用户认知** - TELOS的10个文件让AI真正了解用户
3. **事件驱动的钩子** - 在关键点自动执行操作
4. **清晰的分层架构** - Hot/Warm/Cold三层，便于管理

### 我们的优势

1. **语义搜索** - memory_search可以快速找到相关记忆
2. **时间分层** - P0/P1/P2基于重要性的时间管理
3. **已有的反思机制** - 每日反思章节

### 最佳结合

将PAI的**信号捕获**、**钩子系统**、**结构化用户认知**与我们的**语义搜索**、**时间分层**相结合，创建一个更强大的记忆系统。
