## 一、CLI优先架构的核心理念

### 1.1 为什么选择CLI优先？
- **AI的概率性本质**：AI生成的内容是概率性的，可能出错、可能不一致、可能有幻觉
- **CLI的确定性本质**：命令行工具是确定性的——相同的输入总是产生相同的输出
- **分离关注点**：
  - CLI工具负责"做什么"（执行具体操作）
  - AI负责"决定做什么"（理解需求、编排工具）

### 1.2 核心原则（从PAI项目提取）
1. **想清楚 + prompt > 一切**：好的prompt来自对需求的清晰思考
2. **CLI优先架构**：先构建确定性的CLI工具，再用AI提示层编排
3. **用工程化方式压低AI的不确定性**：
   - 使用模板和一致的模式
   - 用代码的确定性来压低AI的不确定性
   - UNIX哲学：做好一件事、让工具可组合、用文本界面

## 二、我们现有的CLI工具分析

### 2.1 已有的CLI工具列表
| 工具 | 功能 | 确定性 | 输入 | 输出 |
|------|------|--------|------|------|
| `task_manager.py` | 查看任务状态 | ✅ | 无 | 任务状态文本 |
| `start_task.py` | 标记任务为进行中 | ✅ | task_id | 成功/失败状态 |
| `complete_task.py` | 标记任务为完成 | ✅ | task_id | 成功/失败状态 |
| `add_task.py` | 添加新任务 | ✅ | title, description, priority, due | 成功/失败状态 |
| `session_start_hook.py` | 会话开始钩子 | ✅ | 无 | 状态更新文本 |
| `memory_helper.py` | 记忆管理助手 | ✅ | priority, content, date | 成功/失败状态 |
| `update_directory_index.py` | 更新目录索引 | ✅ | 无 | 成功/失败状态 |

### 2.2 我们的工具符合CLI优先架构的哪些方面？
✅ **每个工具只做一件事**：`start_task.py`只负责标记任务为进行中，不负责执行任务  
✅ **文本接口**：所有工具的输入输出都是文本（参数、JSON、stdout/stderr）  
✅ **确定性**：相同的输入总是产生相同的输出  
✅ **可组合**：可以用shell脚本或AI将多个工具组合起来  

### 2.3 我们的工具还有哪些可以改进的地方？
❌ **缺少详细的使用文档**：部分工具还没有完整的README  
❌ **缺少错误处理和退出码**：部分工具在出错时没有明确的退出码  
❌ **缺少shell补全**：没有bash/zsh补全脚本  
❌ **缺少配置文件支持**：部分硬编码的路径可以移到配置文件中

## 三、AI编排层的设计

### 3.1 AI编排层的职责
1. **理解用户需求**：将自然语言需求转换为CLI工具调用序列
2. **规划执行顺序**：决定调用哪些工具、按什么顺序调用
3. **处理输入输出**：将前一个工具的输出作为后一个工具的输入
4. **处理异常和重试**：当工具失败时，决定是重试、跳过还是报告错误
5. **向用户反馈**：将执行结果用自然语言反馈给用户

### 3.2 简单的编排示例

#### 示例1：完成一个任务
**用户需求**："帮我完成task-010，然后更新任务状态"

**AI编排步骤**：
1. 调用 `python3 start_task.py task-010`（标记为进行中）
2. 执行任务的具体工作（可能涉及其他工具）
3. 调用 `python3 complete_task.py task-010`（标记为完成）
4. 调用 `python3 task_manager.py`（查看更新后的状态）
5. 用自然语言向用户反馈："已完成task-010，当前任务状态：..."

#### 示例2：添加并完成一个新任务
**用户需求**："添加一个任务'深入研究CLI优先架构'，优先级P1，然后完成它"

**AI编排步骤**：
1. 调用 `python3 add_task.py "深入研究CLI优先架构" "分析CLI优先架构的核心理念和实践" P1`
2. 获取返回的task_id（假设是task-011）
3. 调用 `python3 start_task.py task-011`
4. 执行研究工作（创建研究笔记等）
5. 调用 `python3 complete_task.py task-011`
6. 向用户反馈
